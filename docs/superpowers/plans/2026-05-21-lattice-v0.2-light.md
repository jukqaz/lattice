# Lattice v0.2 Light Implementation Plan

Status: implemented and superseded by the v0.3.0 release preparation. Release
steps require explicit user approval before commit, tag, push, or GitHub release
creation.

> **For agentic workers:** This is historical implementation evidence. Do not
> commit, tag, push, or create releases unless the user explicitly asks.

**Goal:** Make Lattice safe enough for real personal dotfiles by adding restore conflict detection, snapshots, minimal hooks, secret guardrails, and stronger harness coverage.

**Architecture:** Keep the product lightweight. Extend the existing TOML service model and copy-based operations instead of introducing a database, plugin system, or git sync. Put safety logic in focused library modules and keep CLI code as command orchestration.

**Tech Stack:** Rust 2024, `clap`, `serde`, `toml`, standard library filesystem/process APIs, existing `xtask` harness.

---

## File Structure

- `crates/lattice-core/src/config.rs`: add hook config models.
- `crates/lattice-core/src/hooks.rs`: run or describe command hooks by phase.
- `crates/lattice-core/src/secrets.rs`: lightweight secret-looking content scan.
- `crates/lattice-core/src/ops.rs`: restore conflict detection, snapshots, force behavior, and backup secret scan wiring.
- `crates/lattice-core/src/lib.rs`: export new modules.
- `crates/lattice-cli/src/main.rs`: add `--force`, `--yes`, `--allow-secret-looking-files`, and `validate`.
- `crates/lattice-cli/tests/cli_smoke.rs`: cover conflicts, snapshots, hooks, custom services, and secret scan CLI behavior.
- `xtask/src/main.rs`: add real `~/.codex` read-only dry-run coverage.
- `TODO.md`: mark v0.2 items as implemented only after verification.
